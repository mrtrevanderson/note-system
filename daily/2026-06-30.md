### Declared Audience - Segment Registry Open Decisions
> 12:15-12:45 | source: fellow/2026/2026-06-30/declared-audience-segment-registry-open-decisions.md

The team reviewed open technical decisions for the Segment Registry pipeline planned for Q3 2026, working through gaps in the PRD that need resolution before development begins. Key decisions included using the mailable universe data source (not raw data), implementing a single YAML file approach for tracking segments, and requiring all validations to occur at JIRA ticket creation rather than mid-pipeline. The team agreed to confirm PII attribute requirements with compliance (Jeff Richards and Jason) and contact vendors (TransUnion, Live Ramp, Loadata) about preferred data delivery formats. Additional architectural decisions addressed future multi-vendor distribution through Databricks Identity Hub, the need for analytical tables to enable reporting on segment performance, and implementing a human approval gate in the workflow. Joseph will draft vendor communications and update the Confluence documentation with all decisions made during this meeting.

**Discussion:**
- [01:04] The Segment Registry is a new pipeline being built off the existing legacy declared audience pipeline, with development planned for Q3 2026.
- [01:24] Trevor and Joseph identified open questions and decision items in the PRD organized as 'gaps' A through F.
- [02:18] Gap A addresses whether to pass raw PII or only hashed PII to vendors (TransUnion, Live Ramp, Loadata); legal and compliance review required.
- [04:38] Currently declared audiences send email hash, first name, last name, city, state, address, and zip code; the team needs definitive compliance sign-off.
- [06:04] Gap B resolved: use mailable universe data source rather than raw survey data, avoiding pre-FTC-compliance users.
- [07:21] Gap C resolved: demographic and geographic PII will require joining the user table with the survey event table in mailable universe.
- [10:16] Gap D concerns user-perspective vs segment-perspective vendor delivery output; engineering will draft a technical inquiry to TransUnion, Live Ramp, and Loadata.
- [13:56] Gap F: the team decided on a single registry YAML file containing all segments rather than 2,000+ individual files.
- [16:41] The team confirmed the need for analytical metric tables separate from pipeline tables to enable reporting on segment creation, agent attribution, and vendor performance.
- [20:49] Databricks Identity Hub will enable bi-directional sync of audiences to activation channels (Meta, etc.) starting Q4 2026.
- [24:29] The team confirmed using a Claude skill-based template where Virginia/account managers input requirements, and the skill creates JIRA tickets only after validating all required fields.
- [25:56] Primary validations (including a 1 million record minimum) must occur before JIRA ticket creation to avoid mid-pipeline failures.
- [28:33] The existing monitoring job checking segment record minimums won't be needed for new segments but should run once as a diagnostic on legacy segments.
- [32:12] A human approval stage gate is required in the workflow to prevent automated creation of incorrectly specified segments from freeform text input.
- [34:23] The team will schedule a follow-up meeting when Manav returns (Tuesday or Wednesday of the following week).

**Decisions:**
- The segment registry will use mailable universe as the data source, not raw data, for all segment creation.
- Segment tracking will use a single YAML file registry containing all segments rather than separate files per segment.
- All segment validations (including the 1 million record minimum threshold) must occur at JIRA ticket creation, not mid-pipeline.
- Pipeline output must include vendor delivery information to track which vendors receive each segment.
- A human approval gate is required in the workflow to review and confirm segments before activation.
- Analytical metric tables will be built to enable reporting on segment creation, performance, vendor attribution, and tie-back to intake requests.
