### Joseph x Trevor Sync
> 10:30-11:00 | source: fellow/2026/2026-06-24/joseph-x-trevor-sync.md

Joseph and Trevor discussed significant concerns about the new segment registry implementation being developed by Manov and Dan. Joseph identified multiple technical and process issues, including **unclear PII requirements** (the spec mentions no names, addresses, or phone numbers, contrary to what Virginia likely needs), **performance concerns with querying large tables** (100M users, 1.4B survey records) sequentially for potentially 3,000 segments, and **fundamental differences in output format** (segment-wise vs. the current user-wise approach). The team agreed that Virginia needs to be consulted immediately to clarify requirements before implementation proceeds, as the current design may not align with client SLAs or existing delivery formats. Trevor will coordinate a meeting with Manov to review the architecture, while Joseph will draft a summary of concerns for Virginia by end of day.

**Discussion:**
- [00:00:53] Joseph: segment registry spec indicates no names, addresses, or phone numbers will be sent to clients, which conflicts with current practices where all PII is sent (matching what Virginia requires for similar products like Playful Rewards).
- [00:02:06] Joseph: survey tables contain only email hashes, not full PII; if no-PII approach is followed, users table would not need to be joined, fundamentally changing the architecture.
- [00:02:14] Trevor: unclear how hashed identifiers alone provide value to data marketplace consumers — the whole point of these segments is to buy PII; needs clarification from Virginia.
- [00:03:39] Joseph: team must work with Virginia to confirm requirements and revisit the EPIC before implementation; SLA changes would be required if data format changes from current practices; design was never peer reviewed — Joseph and Grant had no input.
- [00:05:38] Joseph: each segment is defined in separate YAML files (potentially 3,000 files); difficult to locate specific segments without grep commands since only IDs are in filenames.
- [00:07:11] Joseph: current design queries both users table (~100M records) and survey table (~1.4B records) sequentially for each segment — potentially thousands of redundant full table scans with poor performance.
- [00:08:35] Joseph: segment definitions reference raw tables instead of gold tables that Leo produces, deviating from best practices and current compliant pipeline.
- [00:09:25] Joseph: segment size validation shows 1,000 minimum rows in spec, but Virginia's actual requirement is 1 million+ for declared audiences — missing eligibility checks in the design.
- [00:09:44] Joseph: new segment registry produces segment-wise output (one file per segment listing users), while current approach uses user-wise output (one row per user with pipe-separated segment list), creating a fundamental mismatch.
- [00:10:21] Joseph: current delivery format for LiveRamp and TransUnion includes one row per user with email, hashes, PII, and all segments that user belongs to, following established client-specific column mappings (10 columns for LiveRamp, 14 for TransUnion).
- [00:17:17] Joseph: if only survey tables are used without users table, pipeline would miss demographic segments like state, gender, and age that exist only in the users table.
- [00:19:57] Trevor: suggested table-based approach for storing segment metadata (with history/audit capabilities) would be more optimal than reading from 3,000 YAML files; the current DA pipeline already has such a table with ~60 columns.
- [00:24:19] Joseph: discovered the Confluence doc and repo only yesterday; Manov was working in isolation without team review or collaboration.
- [00:26:43] Joseph: requested inclusive planning meetings when new initiatives are started so concerns can be identified before implementation begins.
- [00:27:10] Trevor: design was never peer reviewed by the team; Dan posted it, Manov started working on it independently; gap in collaborative review process.

### In-store - Databricks Schema and Support
> 11:00-11:30 | source: fellow/2026/2026-06-24/in-store-databricks-schema-and-support.md

The meeting focused on designing the **data architecture for a new in-store advertising platform** that will integrate with Databricks and use programmatic bidding through DSPs. Aaron Lupo presented the technical architecture to the data engineering team, explaining how the platform will display ads on POS devices at retailers like Walgreens and CVS, using an **open-source Prebid server to conduct real-time auctions** (within ~100 milliseconds) with multiple DSP partners. The system will capture both authenticated and unauthenticated impressions, with authenticated users able to engage with offers via SMS or email, creating a trackable conversion funnel that doesn't exist in traditional in-store advertising.

The team decided to **follow Pulse's existing pattern of pushing event data into Kafka streams** for Databricks ingestion, allowing for flexibility as requirements evolve. Build (the hardware partner) will deploy the first Verifone tablets to stores on **Monday, June 29, 2026**, initially using Pulse's partner channel API before transitioning to the new OpenRTB-based system post-October 2026. The architecture is designed to scale to **10 million transactions daily** within a year, and the Cybage team will handle implementation while Trevor and Kapil provide design guidance.

**Discussion:**
- [00:00:15] Aaron Lupo: goal is to ingest event data into Databricks in a standardized, cost-effective way, learning from the Pulse implementation.
- [00:02:46] Aaron: environment will be set up in the same cluster as Pulse but in a different namespace — separate tech stacks able to communicate within the cluster.
- [00:03:51] Kapil: from Databricks perspective need two things — events/clickstream data (impressions, engagements) and metadata (campaigns, creatives, partners) which should be API-driven for internal apps.
- [00:05:01] Aaron: consumer experience — POS device displays ad when first item is swiped at checkout; unauthenticated impression shown immediately; authenticated users get second ad request with known hash phone/email, demographics, audience segments.
- [00:06:15] Aaron: platform enables consumer engagement with in-store ads unlike traditional display; user can put in loyalty number to get personalized offers.
- [00:07:47] Aaron: platform will predominantly fill inventory with external DSP demand (Trade Desk, Amazon DSP) rather than self-sourced demand; volume of sessions will exceed available internal inventory.
- [00:10:42] Aaron: unique request ID serves as session identifier (similar to Pulse's offer ID), passed to DSPs and logged throughout entire user journey for attribution.
- [00:11:11] Aaron: system will log all bid responses from DSPs including bid values, advertiser details, and winning bid selection.
- [00:12:59] Aaron: users can engage with offers via 'send me the details' CTA that triggers SMS or email delivery of the offer to their personal device.
- [00:14:13] Aaron: even 1% conversion rates from impression to offer redemption would be valuable — advertisers currently have no engagement tracking for in-store advertising.
- [00:20:38] Jack Hall: integrating through open-source Prebid framework with OpenRTB standard; all connected DSPs bid and entire auction completes in approximately 100 milliseconds.
- [00:23:24] Aaron: Prebid server handles auction logic including bid prioritization, minimum floor prices, request formatting, and user enrichment with data from Fluent identity graph.
- [00:27:25] Aaron: platform is essentially a 'Pulse Lite' — standardized reimagining of what Pulse would look like if rebuilt today, minus ad management and partner management complexity.
- [00:27:32] Aaron: architecture designed to handle 10 million transactions daily by this time next year.
- [00:35:08] Bharat: team could follow same pattern as Pulse — push data into Kafka stream; Databricks ingests from there; flexible as requirements evolve.
- [00:35:49] Team: agreed to start with raw data feed in Databricks since there are unknowns; curate for reporting after analyzing the feed.
- [00:37:00] Aaron: two separate data flows — OpenRTB request/response logging for audit control, and actual ad performance events; need to be treated as distinct data streams.
- [00:38:35] Aaron: Build (hardware partner) is primarily a payment processing company focused on shipping new Verifone tablets; first device live in store on Monday June 29; advertising integration is secondary priority for Build.
- [00:39:30] Aaron: Build using Pulse's partner channel API initially for low volume; new system must be ready for post-October 2026 as current approach doesn't scale beyond Q4.

**Decisions:**
- Event data from the in-store platform will be pushed into Kafka streams following the same pattern as Pulse, with Databricks ingesting from there for downstream analytics and reporting.
- The Cybage team will handle implementation of the data ingestion layer while Trevor and Kapil provide design support and guidance on Databricks integration.
- The in-store platform will run in the same Kubernetes cluster as Pulse but in a different namespace, allowing the platforms to remain separate while communicating within the cluster.
